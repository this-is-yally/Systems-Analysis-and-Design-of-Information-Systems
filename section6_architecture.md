# Розділ 6. Архітектура системи та залежності між підсистемами

## 6.1 Архітектура системи

Система управління бібліотекою побудована за **клієнт-серверною архітектурою** (3-рівнева):

1. **Клієнтський рівень** (веб-інтерфейс)
   - Браузер працює із UI (фронтенд), що реалізує пошук, бронювання, авторизацію.
   - Відповідає за відображення даних та взаємодію з користувачем.

2. **Серверний рівень** (API / бізнес-логіка)
   - Веб-сервер приймає запити (REST/GraphQL) від клієнта.
   - Модуль авторизації/аутентифікації, модуль обліку книг, модуль видачі/повернення, модуль звітності.
   - Реалізує перевірку прав доступу та виконання бізнес-правил.

3. **Рівень зберігання даних** (база даних)
   - Реляційна СУБД з таблицями для книг, користувачів, позик, бронювань, тощо.
   - Забезпечує цілісність, транзакції та швидкий пошук.

## 6.2 Підсистеми та залежності

- **Підсистема каталогу книг** залежить від бази даних (таблиць `Book`, `Author`, `Category`).
- **Підсистема користувачів** (реєстрація/авторизація) залежить від модуля аутентифікації та таблиці `User`.
- **Підсистема видачі/повернення** залежить від каталогу книг, користувачів та журналу позик (`Loan`).
- **Підсистема бронювання** залежить від наявності книг, даних про користувача та розкладу видач.
- **Підсистема звітності** залежить від підсистем каталогу, видачі/повернення та бронювання для створення статистик.

## 6.3 Схема зв'язків (High-level)

- Клієнт → API (HTTP/HTTPS)
- API → База даних (SQL-запити)
- API → Сервіс сповіщень (email, SMS) (за потреби)
